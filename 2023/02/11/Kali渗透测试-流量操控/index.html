流量操控技术

  • Traffic manipulation technique
  • 渗透测试中经常遇到访问受限的网络环境
  • 使用隐蔽的手段逃避安全检查措施和溯源追踪
  • 在非受信任的网络中实现安全的数据传输

重定向（Redirection）

客户端向真实服务器发送请求，将其请求的真实IP和Port重定向到攻击者的IP和Port，之后攻击者再将客户端的请求信息发送给真实服务器；服务器回包同样也是经过攻击者后返回给客户端。缺点是不能提供传输通道的加密安全。

Rinetd

  • 配置文件：/etc/rinetd.conf
  • 应用场景
    • 重定向web流量，突破上网限制
    • 远程桌面重定向
    • NC重定向获得shell
    • 不兼容FTP等二次连接的协议

假设防火墙只允许53端口出站流量，想要请求目标服务器的流量，但会被防火墙拦截；可以在家里安装Rinetd后侦听TCP/53端口，只要访问家用电脑TCP/53端口就会将请求转发给目标服务器；这样就可以在内网环境中访问家用电脑的TCP/53端口重定向到目标主机。

需要在家用电脑中进行重定向的配置

配置Rinetd的配置文件，绑定本地IP（侦听），绑定的本地端口根据防火墙允许出口的端口设置

隧道（Tunneling）

  • 在不受信任的网络环境中实现安全的通信
  • 通常使用多种加密技术建立通信隧道
  • 点到点（IP2IP）、端到端（Port2Port）隧道
  • VPN：PPTP(有些明文传输)、L2TP、IPSec、SSL vpn

SSH 隧道

  • SSH支持双向通信隧道
    • 将其他 TCP 端口的通信通过 SSH 链接来转发
    • 用SSH作为传输层协议，对流量自动加解密
    • 突破防火墙访问规则的限制，可用于翻墙
  • SSH 端口转发
    • 本地端口转发
      • 本地侦听端口，访问转发到远程主机指定端口
      • SSH流量方向与应用流量方向相同
      • 若开通隧道的主机和侦听端口的主机是同一主机就是本地端口转发
    • 远程端口转发
      • 远程侦听端口，访问转发到本机主机指定端口
      • SSH流量方向与应用流量方向相反
      • 若开通隧道的主机与侦听端口的主机不是同一主机就是远程端口转发
    • 动态隧道模式

SSH 本地端口转发

  • 使用效果类似于rinetd
  • 将本地端口与远程服务器建立隧道
  • 本地侦听端口，访问转发到远程主机指定端口
  • SSH客户端 + 应用客户端位于防火墙一端
  • SSH服务端 + 应用服务端位于防火墙另一端

配置远程主机SSH文件

  • vim /etc/ssh/sshd_config

开启服务器web程序

建立隧道连接

  • ssh -L <listen port>:<remote ip>:<remote port> user@<ssh server> -p <ssh server port>

-L：侦听本地端口

remote ip/port：目标应用服务的IP/端口

ssh server/port：建立ssh链接的主机/端口

一般默认端口转发使用侦听端口范围：7000+

此时只要访问本地7001端口，流量就会流到192.168.216.172的80端口

访问本地7001端口

-f：后台运行进程

-N：建立连接，不执行登录shell

-C：连接通信过程进行压缩，节省带宽

此时连接已经建立，但是在后台运行

当远程主机开启Apache服务后，由于本地主机和远程主机建立的SSH隧道连接，可以认为远程主机的80端口开放在本地主机，因此在remote IP字段处可替换为localhost

-g：复用访问时作为网关，支持多主机访问本地侦听端口

SSH 远程端口转发

由于ACL、防火墙规则等原因，SSH与应用建立连接方向相反，此时需要SSH远程端口转发。此时，Transfer运行SSH客户端并侦听端口，通过建立SSH隧道，将LAN主机的访问流量转发到远程主机Server指定端口 (之所以称为远程端口转发，是因为SSH Client和侦听端口不在隧道同一端)

  • SSH客户端、应用客户端位于防火墙两端
  • SSH客户端、应用服务端位于防火墙两端

应用客户端侦听端口

  • ssh -fNg -R <listen port>:<remote ip>:<remote port> user@<SSH server> -p <ssh server port>

此时SSH客户端已经建立SSH隧道连接，但端口是开放在应用客户端（SSH服务端）

查看SSH服务端是否开启侦听

在SSH服务端访问服务

SSH 动态端口转发

  • 本地、远程端口转发都需要固定应用服务器IP、Port

    • 应用端口繁多，逐个转发效率低
    • 某些应用不固定端口
    • 某些网站不支持IP直接访问
    • 使用非受信网络上网时保护流量不被嗅探

  • 本地侦听socks4/5代理端口

    • 由SSH server决定如何转发
    • 作为翻墙代理使用
    • 配置客户端代理（浏览器）
    • 使用አ proxychains支持无代理客户端
    • ssh -CfNg -D 7001 root@1.1.1.1 -p 2121

DNS协议隧道

  • 防火墙禁止TCP出站访问流量
    • SSH隧道、端口转发全部失效
    • 使用基于UDP协议的隧道
    • DNS的工作原理适合用于实现隧道
  • DNS隧道
    • DNS协议工作在TCP/UDP 53端口，UDP端口用于DNS解析请求，TCP端口用于同一个域下DNS服务器数据同步，一般小型局域网都会使用公网DNS服务器作为本地DNS服务器，所以如果防火墙限制严格，完全可以禁止TCP流量出站，此时基于TCP端口的隧道失效（如SSH隧道），需要搭建基于UDP的隧道。一般防火墙至少会开放UDP 53端口，可以利用DNS协议的工作原理实现UDP隧道
  • DNS隧道原理
    • 注册受自己控制的DNS记录
    • DNS隧道实现的前提是指派受自己控制的三级域名服务器 （利用域名服务商提供的域名服务器管理界面，管理所申请域名的主机记录，同时也可以指派下一级子域的域名服务器）。

图中黄色箭头代表指派关系，即本级域名服务器由上一级域名服务器委派。

当最终的域名服务器解析完整的域名后，返回给内网主机域名所对应的IP，再根据IP发送请求包。

Dns2tcp

  • 利用合法DNS服务器实现DNS隧道
  • C/S（dns2tcpc / dns2tcpd）架构
  • 通过TXT记录加密传输数据（A记录长度有限）
  • 隧道建立后保持连接
  • 默认记录生存时间TTL值为 3 秒
  • 利用DNS迭代查询找到受自己控制的DNS服务器，再利用缓存服务器转发客户端具体的查询请求到受控DNS服务器来建立连接

dns2tcpd

  • 服务端配置文件
    • /etc/dns2tcpd.conf
    • .dns2tcprcd
      • 在当前用户的主目录下，创建.dns2tcprcd文件，将所有配置文件内容写入其中，就不需要手动写执行命令
    • 资源可以是其他地址
配置dns2tcpd文件

客户端与服务端建立隧道之后，基于隧道，服务端给客户端提供什么资源；若客户端连接ssh服务，就会连接到服务端的22端口/25端口；这些端口并不是由dns2tcpd打开侦听，而是客户端连接对应端口后，dns3tcpd转发到服务端真实端口上；例如：客户端开启一个侦听端口2222，然后客户端使用ssh去连接2222端口，此时就会通过隧道发到服务端的53端口（隧道出口），之后dns2tcpd提供的资源就会转发到服务端的22端口，至于22端口是哪一个程序在侦听由服务端决定，也就是说：客户端访问ssh（资源）就会被转发到服务端的22端口

配置ssh服务

启动dns2tcpd
  • dns2tcpd -F -d 1 -f /etc/dns2tcpd.conf

-F：前端运行

-d：debug level 1-3

-f：配置文件路径

注意：若在启动dns3tcpd时出现socket.c:107 bind error报错，如下解决方法：

1、普通进程占用53端口
第一步： netstat -anp | grep:53 # 找到占用的53端口
第二步：执行 kill PID 杀死对应进程即可
2、若是systemd-resolved服务占用53端口
第一步：systemctl stop systemd-resolved  # 停止服务
第二步：vim /etc/systemd/resolved.conf   # 编辑配置文件
第三步：按照以下内容修改后保存
[Resolve]
DNS=8.8.8.8  #取消注释，增加dns
#FallbackDNS=
#Domains=
#LLMNR=no
#MulticastDNS=no
#DNSSEC=no
#Cache=yes
DNSStubListener=no  #取消注释，把yes改为no
第四步：ln -sf /run/systemd/resolve/resolv.conf /etc/resolv.conf  # 运行此命令后重启即可

至此服务器端配置完成

Iodine

  • 基于DNS查询的隧道工具

  • 与同类工具相比的优点

    • 对下行数据不进行编码，因此性能优
    • 支持多平台：Linux、BSD（Unix分支）、Mac OS、Windows
    • 最大16个并发连接（建立16个并发的隧道）
    • 强制密码支持
    • 支持同网段隧道IP（不同于服务器、客户端网段）
    • 支持多种DNS记录类型
    • 丰富的隧道质量检测措施

  • iodine隧道：会给隧道两端分别分配一个同一个网段的IP地址，此相同网段不能相同于客户端和服务端使用的网段，相当于在客户端和服务端建立了一条虚拟的网线。

  • 隧道网络接口

    • 不基于资源的通用隧道，如同本网段内两台相邻的主机通信
    • 服务端和客户端分别生成隧道网络接口dns0
    • 隧道两端接口的IP地址应不同于客户端和服务端的IP网段
    • 基于此隧道可嵌套其他隧道技术
